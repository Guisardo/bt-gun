<objective>
Build the Android Bluetooth HID adapter seam and strict output-report mapper.

Purpose: Android must register as a HID device, send input only when a host is connected, and prove whether macOS-origin output reports can trigger bounded phone haptics.

Output: TDD output mapper, `BluetoothHidDevice` adapter/proxy seam, callback status model, and adapter tests.
</objective>

## Phase Goal

**As a** user with the discontinued iPega AR gun, **I want to** use the Android phone as a Bluetooth HID gamepad for macOS Apple Silicon, **so that** macOS sees a normal controller without paid Apple virtual HID entitlements while the completed Windows VHF path remains fallback.

## Artifacts this phase produces

- HID output report mapper compatible with report ID `0x02`, version `0x01`.
- `BluetoothHidDevice` adapter that owns register/unregister/send/reportError/replyReport calls.
- Status model exposing app registration, host connection, last input report, output callback, validation, haptic result, and unsupported reason.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Android Bluetooth host -> app callback | Host-origin output bytes are untrusted input. |
| App status -> user dashboard | Registration and connection states can be misleading if collapsed. |
| Android app -> phone vibrator | Output report can cause physical feedback. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-08 | Tampering | Output report mapper | mitigate | Validate id, type, length, version, ranges, flags, and reserved bytes before haptics. |
| T-07-09 | DoS | Phone haptics | mitigate | Reuse `DesktopHapticCommand` caps for duration and TTL; invalid reports trigger `reportError`. |
| T-07-10 | Spoofing | HID host status | mitigate | Track connected host state from callbacks and prevent sends without registered+connected status. |
